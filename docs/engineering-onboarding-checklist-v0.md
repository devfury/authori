# Engineering Onboarding Checklist v0

목표: 신규 엔지니어가 Day 1 ~ Day 7 내에 로컬 개발, 리뷰, 배포 기본 흐름을 독립적으로 수행한다.

## Day 1: 계정/권한
- GitHub 조직 초대 수락 및 저장소 읽기/쓰기 권한 확인
- 개발 DB/스테이징 DB 접근 권한 요청 및 승인 확인
- 배포 파이프라인(예: CI/CD) 로그 조회 권한 확인
- 비밀정보 전달 채널 확인(환경변수/시크릿 매니저)

## Day 2: 로컬 환경 준비
- Node.js, npm 버전 확인 (`node -v`, `npm -v`)
- 의존성 설치
  - `cd backend && npm ci`
  - `cd ../frontend && npm ci`
- 환경변수 파일 준비 (`backend/.env`)
- DB 연결 확인 및 마이그레이션 실행
  - `cd backend && npm run migration:run`

## Day 3: 서비스 기동/기본 검증
- 백엔드 기동: `cd backend && npm run start:dev`
- 프론트엔드 기동: `cd frontend && npm run dev`
- 로그인/핵심 관리 화면 진입 및 기본 API 정상 응답 확인

## Day 4: 코드 변경/테스트 기본 플로우
- 브랜치 생성 규칙 확인 후 작업 브랜치 생성
- 변경 후 최소 검증
  - `cd backend && npm run build`
  - `cd frontend && npm run build`
- 필요한 단위/통합 테스트 실행

## Day 5: PR/리뷰 절차
- PR 템플릿 기준으로 변경 요약, 영향 범위, 검증 방법 기재
- 스크린샷/로그/재현 절차 첨부
- 리뷰어 지정 후 코멘트 반영, 강제 푸시 최소화

## Day 6: 배포 준비
- 릴리즈 범위와 롤백 전략 문서화
- 스키마 변경 시 마이그레이션 적용 순서 확인
- 운영 반영 체크리스트(환경변수, 배치, 모니터링) 점검

## Day 7: 배포/사후 확인
- 배포 직후 헬스체크 및 핵심 사용자 플로우 점검
- 오류 로그/알람 모니터링 30분 이상 확인
- 이슈 트래커에 결과 및 후속 액션 기록

## 브랜치/PR/리뷰/배포 기본 규칙
- 하나의 PR은 하나의 목적에 집중한다.
- DB 마이그레이션 포함 시, 적용 순서와 롤백 방법을 반드시 PR에 명시한다.
- 리뷰 승인 전 자체 빌드 실패 상태로 PR을 올리지 않는다.
- 배포 후 장애 발생 시 즉시 롤백하고 원인/재발방지 항목을 이슈에 남긴다.

## DoD (CTO 셀프리뷰 기준)
- 체크리스트가 Day 1 ~ Day 7 흐름으로 1페이지 내 요약되어 있다.
- 권한/환경/개발/리뷰/배포 항목이 모두 포함되어 있다.
- 실제 명령 예시가 포함되어 신규 인원이 즉시 실행 가능하다.
